About Amstel

Amstel is a framework for processing large-scale graphs following the model published by Google called Pregel. The motivation behind building such a framework is to hide the complexity of developing parallel and distributed systems under a generic framework that helps its users to focus more on the problem by offering a computational model and mechanisms to express (almost) any graph algorithm. It is somehow similar with the Map-Reduce paradigm, except that it is for large graphs and not for large arrays.

Amstel is implemented in Java, on top of the IPL communication library, and it is primarily intented, but not limited, to be run on DAS3 and DAS4 clusters from Vrije Universiteit Amsterdam.

Computational model

The computational model origins from BSP (Bulk Synchronous Parallel) model published by Leslie G. Valiant in 1990 as an alternative to the von Neumann model for hardware-level computation. The programs are running in a sequence of iterations called super-steps, in which each vertex from the graph can send messages to other vertices, modify its state or read messages received from the previous super-step. The number of super-steps is not known in advance and the system terminates when there are no active vertices.

The computation is centered around the vertex and the user has to provide the compute() method that will be run on each vertex in an unspecified order, in each super-step. The algorithm terminates when there are no active vertices in the system. Initially, all vertices are active. In a given super-step, a vertex can decide to deactivate itself by a voteToHalt() operation and any vertex becomes automatically active when it receives a message from another vertex.

Architecture overview

The distributed system behind Amstel has a centralized architecture, similar with the one described by Pregel. A central node called the Master node is used for splitting the input data, partitioning the graph and for global synchronization during each super-step. Worker nodes keep the assigned partition of the graph in memory and they are responsible for running the compute() method on each vertex, to offer each vertex the messages received in the previous super-step and send the messages for the next super-step efficiently.

The communication primitives used are the ones exposed by IPL and it involves synchronous one-to-many communication between the Master and the Worker nodes, but also asynchronous one-to-one communication between the Worker nodes for delivering messages across super-steps.
